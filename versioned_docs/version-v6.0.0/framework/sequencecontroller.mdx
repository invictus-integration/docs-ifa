:::note[motivation]
Some dependent external systems can't handle parallel message processing or requires a certain order of messages. Sending messages serially (for example, by setting the concurrency on a Logic App to `1`) can be a solution, at the cost of a big performance impact.

<FontAwesomeIcon icon={faEarthEurope} /> **A customer story from the trenches** <br/>
*A downstream application can't handle calls at the same time for the same order ID, by using the **Sequence Controller** we guarantee a single call per order ID (by using the order ID in the `sequenceName`) and we don't have the use the concurrency option for the Logic App which would dramatically slow down the processing for all messages.*

The Invictus Framework provides a **Sequence Controller** component that allows you to process Logic App workflow runs in a specified order. Even parallel triggered workflows won't overblown the external dependency with messages.

<FontAwesomeIcon icon={faLink} /> See also the [Message Sequence](https://www.enterpriseintegrationpatterns.com/patterns/messaging/MessageSequence.html) integration pattern.
:::

## Available endpoints
The **Sequence Controller** component is available as a HTTP endpoint in your Logic App workflow. To include sequence processing to your workflow, add these three HTTP interaction tasks:

* <a href="#-get-sequence-number">`/api/GetSequenceNumber`</a>: allows the workflow to determine what current position it has in the sequence;
* <a href="#-wait-for-sequence">`/api/WaitForSequence`</a>: allows the workflow to wait its turn, doing the actual work in a **Control** task;
* <a href="#-complete-sequence">`/api/ConfirmExecutionCompleted`</a>: allows the workflow to signal that the next workflow in the sequence is up.

<Admonition type="praise" icon={<FontAwesomeIcon icon={faBoltLightning} />} title="Advanced usage: reset sequence">
There also exists a <a href="#-reset-sequence">`/api/ResetSequence`</a> action that allows admins to externally remove references to old sequences or possibly reuse sequence names.
</Admonition>

![Pseudo Logic App workflow with Sequence Controller](/images/framework/pseudo-logic-app-w-sequence-controller.png)

The component processes workflows in sequence after the **Wait**. The place between the **Wait** and **Complete** task allows you to place your own logic that needs to run in order.
If workflow 1 gets triggered before workflow 2, the second workflow waits for the first workflow.

## <FontAwesomeIcon icon={faCircleDown} /> Get sequence number

First step for the Logic App workflow to run in sequence, is to take a number in the line. Doing this requires you to send a HTTP POST request to the `/api/GetSequenceNumber` endpoint of the deployed **Sequence Controller**.

:::tip
You can circumvent this step in some advanced scenarios where you keep track of the order numbers and pass it yourself during the <a href="#-wait-for-sequence">`/api/WaitForSequence`</a>.
:::

The most minimal request body only contains the name of the sequence. 
This correlates all related workflows. Assuming this is the first workflow, it receives number 1 in the response body; the second workflow receives number 2, and so forth.

```json
// POST /api/GetSequenceNumber
{
  "sequenceName": "<my-sequence-name>",
}
```

{/* vale Google.Colons = NO */}
You require the counter in the response body for the next step: <a href="#-wait-for-sequence">`/api/WaitForSequence`</a>.
{/* vale Google.Colons = YES */}

:::tip[start later]
If you want to start the sequence at a later point, you can also pass the `sequenceStart` with your own start number. Only make sure that you pass this along future calls.
```json
// POST /api/GetSequenceNumber
{
  "sequenceName": "<my-sequence-name>",
  "sequenceStart": 10
}
```
:::

## <FontAwesomeIcon icon={faHourglassHalf} /> Wait for sequence

The next step for the Logic App workflow to run in sequence, is to wait its turn. You require the counter collected from the previous <a href="#️-get-sequence-number">`/api/GetSequenceNumber`</a> step.

To make the workflow wait, the workflow requires a HTTP-callback task. This allows you to pass in a webhook that the deployed **Sequence Controller** can call when its the workflow's turn to proceed.

Send this request in this HTTP callback task:

```json
// POST /api/WaitForExecution
{
  "sequenceName": "<my-sequence-name>",
  "counter": @sequenceCounter, // Collected via /api/GetSequenceNumber.
  "callbackUri": @listCallbackUrl() // Available through the HTTP-callback task.
}
```

## <FontAwesomeIcon icon={faCheckDouble} /> Complete sequence

The final step for the Logic App workflow to run in sequence, is to signal the completion of the item in the sequence. This allows the next workflow to proceed. 
You require the counter collected from the previous <a href="#️-get-sequence-number">`/api/GetSequenceNumber`</a> step.

Add a HTTP task with this request to complete the in-sequence work of the workflow:
```json
// POST /api/ConfirmExecutionCompleted
{
  "sequenceName": "<my-sequence-name>",
  "currentCounter": @sequenceCounter, // Collected via /api/GetSequenceNumber
}
```

## <FontAwesomeIcon icon={faRotate} /> Reset sequence

The deployed **Sequence Controller** allows admins to reset stored sequences. This could be useful if you want to reuse certain sequence names or for cleaning references.

:::danger
{/* vale Microsoft.Contractions = NO */}
This action **SHOULD NOT** be part of any Logic App workflow as resetting sequences in the middle of processing will result in indefinitely 'hanging' workflow runs.
{/* vale Microsoft.Contractions = YES */}
:::

Reset the sequence by sending this request:

```json
// POST /api/ResetSequence
{
  "sequenceName": "<my-sequence-name>"
}
```
