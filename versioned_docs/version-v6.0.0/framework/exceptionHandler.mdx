:::note[motivation]
Most who have worked with Logic Apps is familiar with the error `An action failed. No dependent actions succeeded`. This is a generic error that offers no clear information on what went wrong.

Invictus provides an **Exception Handler** Framework component that retrieves the actual error message from an Azure Logic App and optionally translate this to a human readable text.
:::

## Component usage
To centralize exception handling, create a Logic App that acts as a 'fallback handler'. This Logic App calls the Exception Handler component and returns the response. 
All the other Logic Apps can call this fallback Logic App.

:::tip
* Use a [HTTP webhook](https://learn.microsoft.com/en-us/azure/connectors/connectors-native-webhook) to call the fallback Logic App so that if anything fails in the fallback Logic App, you can resubmit this and return the response asynchronously.
* Interact with the component in a scope block and use the **Terminate** action to return the response from the Exception Handler component to the Logic App dashboard.
:::

:::info[Access Rights]
The Exception Handler component requires the role `Logic App Contributor` and additionally `Website Contributor` (for Logic App Standard) to operate correctly. Assign these to the Exception Handler component to allow the component access to the client's Logic Apps.
:::

The **Exception Handler** component has as single endpoint to request inspection of failed Azure Logic App workflow runs.
By default it uses the [Regex Translator](./regextranslation.mdx) component for translating the error, but it can also use Azure AI Foundry if you set the <a href="?q=useAIExceptionHandlerInterpreter#related-bicep-template-parameters">useAIExceptionHandlerInterpreter</a> parameter to `true`.

:::note
The ExceptionHandler function makes use of the [RegexTranslator](./regextranslation.mdx) component to translate the original error from the Logic App.
:::

A solution has an Azure Logic App that used to return the generic `An action failed. No dependent actions succeeded` error. By using the Exception Handler, you can improve this to the actual error which is `Http request failed with status code 'HostNotFound' and status message: 'No such host is known.'.`.

Business users might not understand this message. Use the translation capabilities of the Exception Handler function to translate the error into a clearer message, such as `Application is not reachable`.

Set these translations in the `RegexTranslator` table in your Invictus storage account. Add the translations in the table as follows:

| PartitionKey              | RowKey     | MatchPattern                           | OutputPattern                                           |
| ------------------------- | ---------- | -------------------------------------- | ------------------------------------------------------- |
| `logicappname.actionname` | Any RowKey | The pattern to be matched in the error | code(YourTranslationCode)The output translation pattern |

<details>
<summary>**Translation examples**</summary>

| PartitionKey                                     | RowKey | MatchPattern                                                                      | OutputPattern                                                  |
| ------------------------------------------------ | ------ | --------------------------------------------------------------------------------- | -------------------------------------------------------------- |
| `cdt-dev-we-somelogicapp.CallHTTP`               | 1      | `No such host is known`                                                           | `code(Availability)Application is not reachable`               |
| `cdt-dev-we-somelogicapp.Create`                 | 1      | `cannot be evaluated because array index '0' cannot be selected from empty array` | `code(Account)The accountid for the debtor could not be found` |
| `cdt-dev-we-somelogicapp.ParseJSON`              | 1      | `Invalid JSON schema type: int`                                                   | `code(Data)The field is not an integer`                        |
| `cdt-dev-we-somelogicapp.ExecuteStoredProcedure` | 1      | `Procedure or function 'upsertData' expects parameter '@Ordernumber'`             | `code(Data)Ordernumber is not supplied but is required`        |

By using the translation capabilities of the Exception Handler function, it sends back the following response:

``` json
{
   "status": "Failed",
    "code": "Availability",
    "description": "Application is not reachable",
    "errors": [
        {
            "code": "Availability",
            "description": "Application is not reachable"
        }
    ]
}
```

<details>
<summary>**Ignore Azure Logic App errors**</summary>

By using the translation capabilities of the Exception Handler function, you can also ignore errors. 
This can be useful for scenarios where Logic Apps generates an irrelevant error to the business while the flow ran successfully. 

An example: a solution with an Azure Logic App picks up a file from a network location. After publishing of the file, it deletes the file. 
In some situations this delete action fails because another instance of the Logic App already removed this file. 
You can ignore this error since the message processing already happened.

To achieve this you can configure the translation as follows:

| PartitionKey                         | RowKey | MatchPattern                 | OutputPattern |
| ------------------------------------ | ------ | ---------------------------- | ------------- |
| `cdt-dev-we-somelogicapp.DeleteFile` | 1      | `There is no file with name` | ignore        |

This gives the following response:

``` json
{
    "status": "Succeeded",
    "code": "NoCodeSpecified",
    "description": "",
    "errors": []
}
```
